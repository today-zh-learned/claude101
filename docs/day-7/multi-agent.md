# 멀티에이전트 아키텍처

## 오늘의 목표

> 여러 에이전트가 협업하는 아키텍처 패턴을 이해하고, 상황에 맞는 패턴을 선택할 수 있게 되기

앞 페이지에서 멀티에이전트 시스템의 실전 사례를 봤습니다. 그런데 에이전트를 그냥 많이 만든다고 좋은 시스템이 되는 건 아닙니다.

**어떻게 협업하게 할 것인가** — 이것이 아키텍처의 핵심입니다.

---

## 단일 에이전트의 한계

에이전트 하나로 모든 것을 처리하면 어떻게 될까요?

`"블로그 글 써줘. 키워드 분석도 하고, SEO 최적화도 하고,
 이미지 프롬프트도 만들고, SNS 포스트도 만들어줘."`
결과는 이렇습니다.

| 문제 | 설명 |
| --- | --- |
| **컨텍스트 오버플로** | 지시사항이 너무 많아 앞부분을 잊어버립니다 |
| **품질 하락** | 한 가지에 집중하지 못해 모든 결과가 평범해집니다 |
| **디버깅 불가** | 어디서 문제가 생겼는지 파악하기 어렵습니다 |
| **재사용 불가** | 키워드 분석만 따로 쓸 수 없습니다 |

사람도 마찬가지입니다. 한 사람이 기획, 디자인, 개발, 마케팅을 전부 하면 품질이 떨어집니다. 역할을 나누면 각자 전문성을 발휘합니다.

---

## 3가지 멀티에이전트 패턴

실전에서 자주 쓰이는 3가지 패턴을 소개합니다.

### 패턴 1: 전문가 패널

각 에이전트가 **자기 전문 분야**를 담당하고, 오케스트레이터가 필요에 따라 호출합니다.

`              오케스트레이터
              /     |     \
             /      |      \
     보안전문가  성능전문가  UX전문가`
**적합한 상황**: 코드 리뷰, 다각도 분석, 의사결정 지원

**실제 예시 — 코드 리뷰 시스템**:

`# .claude/agents/security-reviewer.md
당신은 보안 전문 리뷰어입니다.
코드에서 보안 취약점만 집중적으로 찾아주세요.
- SQL Injection
- XSS
- 인증/인가 누락
 
# .claude/agents/performance-reviewer.md
당신은 성능 전문 리뷰어입니다.
코드에서 성능 이슈만 집중적으로 찾아주세요.
- N+1 쿼리
- 불필요한 재렌더링
- 메모리 누수`
오케스트레이터가 PR을 받으면, 보안 리뷰어와 성능 리뷰어를 각각 호출하고 결과를 종합합니다.

### 패턴 2: 파이프라인

에이전트가 **순차적으로** 작업을 전달합니다. 앞 단계의 출력이 다음 단계의 입력이 됩니다.

`리서처 → 기획자 → 작성자 → 리뷰어 → 최종본`
**적합한 상황**: 콘텐츠 제작, 데이터 처리, 단계적 변환

**실제 예시 — 콘텐츠 제작 파이프라인**:

`researcher
  └─ 조사 보고서 → planner
                     └─ 기획안 → writer
                                   └─ 초안 → reviewer
                                                └─ 최종본`
각 단계의 산출물이 명확하기 때문에, 중간에 문제가 생기면 해당 단계만 다시 실행하면 됩니다.

### 패턴 3: 팬아웃/팬인

여러 에이전트가 **동시에** 작업하고, 결과를 한곳에서 합칩니다.

`                분배기
              /   |   \
             /    |    \
         Agent  Agent  Agent    ← 팬아웃 (병렬 실행)
             \    |    /
              \   |   /
                합산기           ← 팬인 (결과 통합)`
**적합한 상황**: 소재 발굴, 시장 조사, 대량 처리

**실제 예시 — 시장 조사**:

`              오케스트레이터
              /     |     \
             /      |      \
       트렌드     키워드    경쟁사       ← 3개가 동시에 실행
       수집       분석      분석
             \      |      /
              \     |     /
             결과 통합/검증              ← 3개 결과를 합쳐서 판단`
3개 작업을 동시에 돌리면 순차적으로 할 때보다 시간이 3분의 1로 줄어듭니다.

실제 시스템에서는 이 3가지 패턴을 **섞어서** 씁니다. 예를 들어 소재 발굴은 팬아웃/팬인, 대본 작성은 파이프라인, 전체 구조는 오케스트레이터 패턴으로 조합할 수 있습니다.

---

## 패턴 선택 가이드

어떤 상황에 어떤 패턴을 쓸지 정리했습니다.

| 상황 | 추천 패턴 | 이유 |
| --- | --- | --- |
| 같은 대상을 여러 관점으로 분석 | 전문가 패널 | 각 관점의 독립성 보장 |
| 작업에 명확한 순서가 있음 | 파이프라인 | 앞 단계 결과가 다음 입력 |
| 독립적인 작업을 빠르게 처리 | 팬아웃/팬인 | 병렬 처리로 시간 단축 |
| 복잡한 시스템 | 혼합 | 단계별로 적합한 패턴 조합 |

## 에이전트 정의하기

Claude Code에서 에이전트를 정의하는 방법입니다. `.claude/agents/` 디렉토리에 마크다운 파일을 만듭니다.

`# .claude/agents/data-analyst.md
 
## 역할
당신은 데이터 분석 전문가입니다.
CSV, JSON 데이터를 분석하고 인사이트를 도출합니다.
 
## 입력
- 분석할 데이터 파일 경로
- 분석 목적 (트렌드, 이상치, 비교 등)
 
## 출력
- 분석 요약 (핵심 인사이트 3-5개)
- 시각화 추천
- 추가 분석 제안
 
## 규칙
- 수치는 반드시 원본 데이터에서 확인할 것
- 추정은 "추정" 표시를 할 것
- 결측치가 있으면 반드시 보고할 것`
핵심은 **역할, 입력, 출력, 규칙**을 명확히 정의하는 것입니다. 모호하면 결과도 모호해집니다.

## Extended Thinking 활용

복잡한 추론이 필요한 에이전트에는 Extended Thinking을 활용합니다.

`# Extended Thinking 모드로 실행
claude --model claude-sonnet-4-20250514 \
  "이 코드베이스의 아키텍처를 분석하고
   리팩토링 계획을 세워줘"`
Extended Thinking은 Claude가 답변 전에 **깊이 생각하는 시간**을 갖게 합니다. 다음 상황에서 유용합니다.

| 상황 | 왜 Extended Thinking인가 |
| --- | --- |
| 복잡한 코드 분석 | 여러 파일 간 의존성 파악 필요 |
| 아키텍처 설계 | 트레이드오프 검토 필요 |
| 버그 진단 | 여러 가능성을 순차적으로 배제 |
| 에이전트 간 충돌 해결 | 상반된 결과를 종합 판단 |

Extended Thinking은 일반 모드보다 토큰을 더 많이 사용합니다. 간단한 작업에는 일반 모드를, 복잡한 추론에만 Extended Thinking을 쓰는 것이 비용 효율적입니다.

## 실습: 3에이전트 파이프라인 만들기

간단한 파이프라인을 직접 만들어봅시다.

`리서처 → 요약가 → 작성자`
**Step 1**: 에이전트 3개 정의

`mkdir -p .claude/agents`
`# .claude/agents/researcher.md
당신은 리서처입니다. 주어진 주제에 대해 핵심 정보를 조사하고
구조화된 보고서로 정리합니다. 출력은 반드시 마크다운 형식입니다.
 
# .claude/agents/summarizer.md
당신은 요약 전문가입니다. 리서처의 보고서를 받아
핵심 포인트 5개로 요약합니다. 각 포인트는 한 문장입니다.
 
# .claude/agents/writer.md
당신은 블로그 작성자입니다. 요약된 핵심 포인트를 바탕으로
독자 친화적인 블로그 글을 작성합니다.`
**Step 2**: 오케스트레이터 역할은 CLAUDE.md에 정의

`# CLAUDE.md에 추가
 
## 블로그 제작 파이프라인
1. researcher 에이전트로 주제 조사
2. summarizer 에이전트로 핵심 요약
3. writer 에이전트로 블로그 글 작성
각 단계의 결과는 output/ 디렉토리에 저장합니다.`
**Step 3**: 실행

`claude "AI 코딩 도구 트렌드에 대해 블로그 글 써줘"`
CLAUDE.md에 파이프라인이 정의되어 있으므로, Claude Code가 순서대로 에이전트를 호출합니다.

## 정리

- 단일 에이전트는 복잡한 작업에서 컨텍스트 오버플로, 품질 하락 문제가 생깁니다

- **전문가 패널**: 다각도 분석에 적합합니다

- **파이프라인**: 순차적 작업에 적합합니다

- **팬아웃/팬인**: 병렬 처리에 적합합니다

- 실제 시스템에서는 이 패턴들을 혼합해서 사용합니다

- 에이전트 정의는 역할, 입력, 출력, 규칙을 명확히 하는 것이 핵심입니다
