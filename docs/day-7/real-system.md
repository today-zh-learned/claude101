# 실전 사례: 콘텐츠 자동화 시스템

## 오늘의 목표

> 실제 운영 가능한 AI 멀티에이전트 시스템의 구조를 이해하고, 나만의 시스템 아이디어 얻기

Day 0에서 Claude Code를 처음 설치했습니다. Day 6에서 에이전트를 만들어봤습니다. 이제 마지막 날입니다. 오늘은 **실전에서 활용할 수 있는 멀티에이전트 시스템**을 설계해봅니다.

예시로 “유튜브 영상 제작 자동화 시스템”을 함께 만들어보겠습니다.

---

## 문제 정의

유튜브 영상 하나를 만들려면 생각보다 많은 단계를 거칩니다.

`소재 찾기 → 키워드 분석 → 리서치 → 기획 → 대본 작성 → 리뷰 → 후처리`
매번 수작업으로 하면 시간이 오래 걸리고, 빠뜨리는 단계도 생깁니다. 이걸 에이전트로 자동화하면 어떨까요?

---

## 시스템 설계

핵심 아이디어는 간단합니다. **각 단계를 전문 에이전트로 만들고, 오케스트레이터가 조율**합니다.

`                  orchestrator (총괄)
                        |
        ┌───────────────┼───────────────┐
        |               |               |
   [소재 발굴]      [기획/작성]       [후처리]
        |               |               |
   ┌────┼────┐     ┌────┼────┐     ┌────┼────┐
   |    |    |     |    |    |     |    |    |
 트렌드 키워드 경쟁  기획  대본  리뷰  숏츠  SNS  데모
 수집   분석  분석  작성  작성       추출 포스트 준비`
### 에이전트 역할 분담

| 단계 | 에이전트 | 하는 일 |
| --- | --- | --- |
| **소재 발굴** | trend-collector | 최신 트렌드 수집 |
|  | keyword-researcher | 검색량 높은 키워드 발굴 |
|  | competitor-analyzer | 경쟁 채널 인기 영상 분석 |
| **기획** | script-planner | 영상 구조 설계 |
|  | hook-generator | 오프닝 훅 후보 생성 |
| **작성** | script-writer | 대본 초안 작성 |
|  | script-reviewer | 대본 리뷰 및 피드백 |
| **후처리** | shorts-extractor | 숏츠용 구간 추출 |

처음부터 이 모든 에이전트를 만들 필요는 없습니다. 대본 작성 에이전트 하나로 시작해서, 필요할 때 하나씩 추가하면 됩니다.

---

## 만드는 방법: 3가지 핵심

### 1. CLAUDE.md — 시스템의 뇌

프로젝트 루트의 `CLAUDE.md`가 전체 시스템 규칙을 정의합니다.

`# CLAUDE.md
 
## 영상 제작 워크플로우
1. 소재 발굴 → 기획 → 승인 → 대본 작성 → 리뷰 → 후처리
2. 기획안은 반드시 사용자 승인 후 대본 작성 진행
3. 산출물은 output/ 디렉토리에 저장`
Day 2에서 배운 CLAUDE.md의 확장판입니다. 규모만 커졌을 뿐, 원리는 같습니다.

### 2. 에이전트 정의 — 각자의 역할

`.claude/agents/` 디렉토리에 각 에이전트를 정의합니다.

`# .claude/agents/script-planner.md
 
## 역할
영상 기획 전문가. 리서치 결과를 바탕으로 영상 구조를 설계합니다.
 
## 입력
- 키워드, 리서치 보고서
 
## 출력
- 기획안 (영상 구조, 섹션별 시간 배분, 훅/클로징)
 
## 규칙
- 영상 길이는 10-15분 기준
- 첫 30초 훅이 가장 중요
- 시청 유지율을 고려한 구성`
### 3. 오케스트레이터 — 지휘자

오케스트레이터가 어떤 에이전트를 언제 호출할지 판단합니다.

`사용자: "MCP 튜토리얼 영상 만들어줘"

오케스트레이터의 판단:
  1. 이미 주제가 있으니 소재 발굴은 건너뜀
  2. 리서치 에이전트 호출
  3. 기획 에이전트 호출
  4. 사용자에게 기획안 보여주고 승인 대기
  5. (승인 후) 대본 작성 에이전트 호출`
---

## 핵심 교훈 4가지

### 1. 작게 시작해서 점진적으로 확장

`v1: script-writer 1개
v2: + script-planner, hook-generator
v3: + trend-collector, keyword-researcher
...
필요에 따라 계속 추가`
### 2. 에이전트마다 역할을 명확히

대본 작성과 리뷰를 같은 에이전트가 하면, 자기가 쓴 글의 문제를 못 찾습니다. 역할을 분리하면 품질이 올라갑니다.

### 3. 오케스트레이터가 전체를 조율

에이전트가 많아지면 누가 언제 뭘 할지 혼란스러워집니다. 오케스트레이터 하나가 교통정리를 해야 시스템이 돌아갑니다.

### 4. 사람의 판단이 들어갈 자리를 남겨라

기획 승인처럼, AI가 전부 결정하지 않습니다. **중요한 결정 포인트에서 사람이 개입**할 수 있어야 합니다.

처음부터 완벽한 시스템을 설계하려 하지 마세요. 에이전트 하나로 시작해서, 필요할 때 하나씩 추가하는 것이 가장 현실적인 방법입니다.

---

## 다른 도메인에도 적용 가능

같은 원리로 다양한 분야에 멀티에이전트 시스템을 만들 수 있습니다.

| 도메인 | 오케스트레이터 | 에이전트 예시 |
| --- | --- | --- |
| **블로그 운영** | blog-orchestrator | keyword-finder, outline-writer, draft-writer, seo-checker |
| **이커머스** | product-orchestrator | market-researcher, listing-writer, price-analyzer, review-monitor |
| **고객 지원** | support-orchestrator | ticket-classifier, faq-matcher, response-writer, escalation-checker |
| **코드 품질** | code-orchestrator | security-scanner, performance-profiler, test-generator, doc-writer |

어떤 도메인이든 패턴은 같습니다.

`1. 반복되는 작업을 찾는다
2. 각 작업을 에이전트로 정의한다
3. 오케스트레이터로 연결한다
4. 사람의 판단 포인트를 넣는다`
“에이전트가 수십 개나 필요할까?” — 아닙니다. 3개면 충분한 경우가 대부분입니다. 중요한 것은 개수가 아니라, **역할 분리와 흐름 설계**입니다.

## 정리

- 반복적인 워크플로우를 에이전트로 분리하면 자동화할 수 있습니다

- CLAUDE.md + 에이전트 정의 + 오케스트레이터, 이 3가지가 시스템의 핵심입니다

- 에이전트 하나로 시작해서 점진적으로 확장하는 것이 현실적인 접근법입니다

- 중요한 결정 포인트에는 반드시 사람의 판단이 들어갈 자리를 남겨야 합니다
