# 나만의 명령어 만들기

## 오늘의 목표

> 자주 하는 작업을 커스텀 슬래시 커맨드로 만들기

Claude Code에는 `/help`, `/quit` 같은 기본 명령어가 있습니다. 그런데 **나만의 명령어**를 직접 만들 수도 있다는 걸 아시나요?

매번 같은 말을 반복하는 자신을 발견했다면, 커스텀 커맨드를 만들 때입니다.

---

## 커스텀 커맨드란?

Claude Code에서 `/`로 시작하는 명령어를 **슬래시 커맨드**라고 합니다.

기본 커맨드는 Claude Code가 제공하는 것이고, 커스텀 커맨드는 **내가 직접 만드는 것**입니다.

`기본 커맨드:   /help, /quit, /clear  ← Claude Code 제공
커스텀 커맨드: /review-kr, /test-all ← 내가 만든 것`
커스텀 커맨드의 정체는 **미리 작성해둔 프롬프트**입니다. 긴 지시를 매번 타이핑하는 대신, 짧은 명령어 하나로 호출하는 거죠.

---

## 폴더 구조

커스텀 커맨드는 `.claude/commands/` 폴더에 Markdown 파일로 만듭니다.

`프로젝트/
├── .claude/
│   └── commands/
│       ├── review-kr.md      → /review-kr 로 호출
│       ├── test-all.md       → /test-all 로 호출
│       └── deploy-check.md   → /deploy-check 로 호출
└── src/
    └── ...`
규칙은 간단합니다:

| 항목 | 규칙 |
| --- | --- |
| 파일 위치 | `.claude/commands/` 폴더 안 |
| 파일 형식 | Markdown (`.md`) |
| 명령어 이름 | 파일 이름이 곧 명령어 이름 |
| 호출 방법 | `/파일이름`으로 호출 |

> ℹ️ **정보**
>
> 폴더가 없으면 직접 만들면 됩니다. `mkdir -p .claude/commands` 명령어로 한 번에 만들 수 있습니다.

---

## 실습 1: /review-kr — 한국어 코드 리뷰

코드 리뷰를 받고 싶을 때 매번 “한국어로 리뷰해줘, 버그도 찾아주고, 개선점도 알려줘…”라고 쓰기 번거롭습니다. 커맨드로 만들어봅시다.

`.claude/commands/review-kr.md` 파일을 만듭니다:

`현재 Git에서 변경된 파일들을 리뷰해주세요.
 
다음 기준으로 검토합니다:
 
1. **버그 가능성**: 런타임 에러가 날 수 있는 부분
2. **보안 문제**: 하드코딩된 비밀번호, API 키 노출 등
3. **코드 품질**: 중복 코드, 불필요한 복잡성
4. **네이밍**: 변수/함수 이름이 의미를 잘 전달하는지
 
결과는 한국어로 작성하되, 코드 예시를 포함해주세요.
각 항목에 심각도를 표시합니다: 🔴 심각 / 🟡 주의 / 🟢 제안`
이제 Claude Code에서 `/review-kr`이라고 입력하면, 이 긴 지시가 자동으로 전달됩니다.

`> /review-kr

# Claude가 자동으로 위 프롬프트를 실행합니다
# Git 변경 사항을 읽고 한국어로 리뷰 결과를 보여줍니다`
---

## 실습 2: /test-all — 테스트 전체 실행

테스트를 돌리고 결과를 정리해달라는 커맨드를 만들어봅시다.

`.claude/commands/test-all.md` 파일을 만듭니다:

`프로젝트의 모든 테스트를 실행해주세요.
 
실행 순서:
1. `npm test` 또는 `pytest` 등 프로젝트에 맞는 테스트 명령어를 찾아 실행
2. 결과를 다음 형식으로 정리
 
## 테스트 결과 요약
- 전체: X개
- 성공: X개
- 실패: X개
 
## 실패한 테스트 (있다면)
- 테스트 이름: 실패 원인 간단 설명
 
## 제안
- 실패한 테스트를 고칠 방법을 제안해주세요`
실행 방법:

`> /test-all`
한 줄이면 끝입니다. 매번 “테스트 돌려줘, 결과 정리해줘, 실패하면 원인 알려줘”라고 쓸 필요가 없습니다.

---

## $ARGUMENTS 변수 사용법

커맨드를 호출할 때 **추가 정보를 넘기고 싶을 때**가 있습니다. 이때 `$ARGUMENTS`를 씁니다.

예를 들어 특정 파일만 리뷰하는 커맨드를 만들어봅시다.

`.claude/commands/review-file.md`:

`다음 파일을 집중적으로 리뷰해주세요: $ARGUMENTS
 
리뷰 기준:
1. 이 파일의 역할과 구조를 먼저 설명
2. 버그 가능성 확인
3. 개선할 점 제안
 
한국어로 작성해주세요.`
호출할 때 파일 경로를 함께 넘깁니다:

`> /review-file src/utils/parser.js`
`$ARGUMENTS` 자리에 `src/utils/parser.js`가 들어갑니다. 하나의 커맨드로 어떤 파일이든 리뷰할 수 있게 되는 거죠.

> ℹ️ **정보**
>
> `$ARGUMENTS`는 슬래시 커맨드 뒤에 오는 모든 텍스트를 받습니다. 파일 경로뿐 아니라 어떤 텍스트든 넘길 수 있습니다.

## 프로젝트 공유 vs 개인 커맨드

커맨드를 어디에 두느냐에 따라 공유 범위가 달라집니다.

| 위치 | 범위 | Git에 포함 |
| --- | --- | --- |
| `.claude/commands/` | 이 프로젝트만 | Yes — 팀원과 공유 가능 |
| `~/.claude/commands/` | 내 모든 프로젝트 | No — 나만 사용 |

**프로젝트 커맨드** (`.claude/commands/`)는 Git에 커밋하면 팀원도 같은 명령어를 사용할 수 있습니다. 코드 리뷰 기준, 배포 체크리스트 같은 **팀 규칙**을 커맨드로 만들면 유용합니다.

**개인 커맨드** (`~/.claude/commands/`)는 내 홈 디렉토리에 저장되어 어떤 프로젝트에서든 쓸 수 있습니다. “내 스타일로 커밋 메시지 써줘” 같은 **개인 취향** 커맨드에 적합합니다.

`# 개인 커맨드 폴더 만들기
mkdir -p ~/.claude/commands`
> ⚠️ **주의**
>
> 프로젝트 커맨드에 회사 내부 정보나 개인 API 키를 넣지 마세요. Git에 커밋되면 다른 사람도 볼 수 있습니다.

---

## 실전 아이디어: 이런 커맨드를 만들어보세요

어떤 커맨드를 만들면 좋을지 감이 안 오시면, 이런 것들을 참고하세요:

| 커맨드 | 용도 |
| --- | --- |
| `/commit-kr` | 한국어로 커밋 메시지 자동 작성 |
| `/explain` | 현재 프로젝트 구조를 설명 |
| `/todo` | 코드에서 TODO 주석을 모두 찾아 정리 |
| `/refactor` | 지정한 파일을 리팩토링 |
| `/doc` | 함수에 JSDoc/docstring 추가 |

자주 반복하는 작업이 있다면, 그게 바로 커맨드로 만들기 좋은 후보입니다.

---

## 정리

오늘 배운 것을 정리합니다:

- 커스텀 커맨드는 `.claude/commands/` 폴더에 Markdown 파일로 만듭니다

- 파일 이름이 곧 `/명령어` 이름이 됩니다

- `$ARGUMENTS`로 호출 시 추가 정보를 넘길 수 있습니다

- 프로젝트 커맨드는 팀과 공유 가능, 개인 커맨드는 나만 사용

- 반복하는 작업이 있다면 커맨드로 만들어보세요
