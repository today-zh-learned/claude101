# 테스트 자동화

## 오늘의 목표

> Claude Code로 테스트 코드를 자동으로 만들고 실행하기

코드를 작성하는 것만큼 중요한 게 있습니다. **그 코드가 제대로 동작하는지 확인하는 것**입니다. Claude Code를 사용하면 테스트 작성과 실행을 거의 자동으로 처리할 수 있습니다.

---

## 테스트가 왜 중요한지

비유를 하나 해봅시다. 자동차에 타기 전에 안전벨트를 매는 것과 같습니다.

안전벨트 없이도 운전은 됩니다. 하지만 문제가 생겼을 때 피해가 큽니다. 테스트도 마찬가지입니다. 테스트 없이도 코드는 돌아갑니다. 하지만 나중에 수정했을 때 **다른 곳이 조용히 망가지는** 일이 생깁니다.

`테스트 없이 코드 수정:
  기능 A 수정 → 기능 B가 망가짐 → 한참 뒤에 발견 → 원인 추적 어려움

테스트 있을 때:
  기능 A 수정 → 테스트 실행 → "기능 B 테스트 실패!" → 바로 발견 → 바로 수정`
특히 Claude Code로 코드를 많이 생성하게 되면, 테스트의 중요성은 더 커집니다. AI가 만든 코드가 정말 의도대로 동작하는지 **자동으로 검증**할 수 있어야 합니다.

프로그래밍 경험이 적더라도 걱정 마세요. 테스트 코드 작성 자체를 Claude에게 맡길 수 있습니다.

---

## ”이 함수에 테스트 만들어줘” 실습

가장 간단한 방법부터 해봅시다. Claude에게 직접 부탁하는 것입니다.

### Step 1: 테스트할 코드 준비

먼저 간단한 함수가 있다고 합시다. 직접 만들어도 되고, Claude에게 만들어달라고 해도 됩니다.

`> 두 숫자를 더하는 add 함수와, 배열에서 최대값을 찾는 findMax 함수를 utils.js에 만들어줘`
Claude가 `utils.js`에 `add`와 `findMax` 함수를 만들어줍니다.

### Step 2: 테스트 만들어달라고 하기

`> utils.js에 있는 함수들에 대한 테스트를 만들어줘`
Claude가 테스트 파일을 자동으로 생성합니다. “두 양수를 더한다”, “빈 배열이면 null을 반환한다” 같은 다양한 케이스를 자동으로 만들어줍니다.

### Step 3: 테스트 실행

`> 테스트 실행해줘`
Claude가 `npm test` 또는 적절한 명령어를 실행하고 결과를 보여줍니다.

테스트 프레임워크(Jest, Vitest 등)가 설치되어 있어야 합니다. 설치가 안 되어 있으면 Claude에게 “Jest 설치해줘”라고 말하면 됩니다.

---

## TDD 스타일: 테스트 먼저, 코드 나중

좀 더 체계적인 방법이 있습니다. **테스트를 먼저 만들고, 그 테스트를 통과하는 코드를 나중에 만드는 것**입니다. 이걸 TDD(Test-Driven Development)라고 합니다.

Claude Code와 함께하면 TDD가 훨씬 쉬워집니다.

### 순서

`1. "이메일 검증 함수의 테스트를 먼저 만들어줘" ← 테스트 먼저
2. Claude가 테스트 파일 생성
3. "이 테스트를 통과하는 함수를 만들어줘" ← 코드 나중
4. Claude가 함수 작성
5. "테스트 돌려줘" ← 확인`
실제로 해봅시다:

`> 이메일 주소가 유효한지 검사하는 함수의 테스트를 먼저 만들어줘.
> 이런 케이스를 포함해줘:
> - 정상 이메일 (user@example.com)
> - @ 없는 경우
> - 도메인 없는 경우
> - 빈 문자열`
Claude가 테스트를 먼저 만들고, 이후 “이 테스트를 통과하는 함수를 만들어줘”라고 하면 함수를 작성합니다.

이 방식의 장점은 **요구 사항이 먼저 명확해진다**는 것입니다. “이런 경우에 이렇게 동작해야 한다”를 테스트로 정의하고, 그에 맞춰 코드를 만드니까요.

---

## Hooks로 커밋 전 자동 테스트 실행

앞에서 배운 Hooks를 활용해봅시다. 커밋할 때마다 **자동으로 테스트를 실행**하게 만들 수 있습니다.

`.claude/settings.json`에 추가합니다:

`{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hook": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'git commit'; then npm test 2>&1 | tail -5; fi"
      }
    ]
  }
}`
이렇게 하면:

`Claude에게 "커밋해줘" → Git 커밋 실행 → 자동으로 테스트 실행 → 결과 표시`
테스트가 실패하면 바로 알 수 있습니다. “커밋은 했는데 테스트가 깨져 있었다”는 상황을 방지합니다.

이것은 Hooks 페이지에서 배운 내용의 실전 응용입니다. Hooks와 테스트를 조합하면 **자동 품질 관리 시스템**이 만들어집니다.

---

## 테스트 실패하면 Claude에게 고쳐달라고 하기

테스트가 실패했을 때가 Claude Code의 진짜 실력이 드러나는 순간입니다.

`> 테스트 실행해줘`
결과:

`FAIL  utils.test.js
  ● findMax › 빈 배열이면 null을 반환한다
    Expected: null
    Received: -Infinity`
이런 실패가 나왔다면:

`> 이 테스트 실패를 고쳐줘`
Claude가 원인을 분석하고 코드를 수정합니다. 이 경우 빈 배열 체크가 빠져 있었다는 걸 찾아내고 고쳐줍니다.

이 과정을 반복하면 됩니다:

`테스트 실행 → 실패 발견 → Claude가 수정 → 테스트 재실행 → 성공`
전부 자연어로 대화하면서 진행할 수 있습니다. 디버깅에 시간을 쏟을 필요가 없습니다.

---

## 테스트 작성 요청 팁

Claude에게 더 좋은 테스트를 만들어달라고 할 때 유용한 표현들입니다:

| 요청 | 효과 |
| --- | --- |
| ”엣지 케이스도 포함해줘” | 빈 배열, null, 음수 등 경계 조건 테스트 추가 |
| ”에러 상황 테스트도 만들어줘” | 잘못된 입력에 대한 에러 처리 테스트 |
| ”현실적인 테스트 데이터로 해줘” | foo, bar 대신 실제 데이터로 |
| ”테스트 커버리지 확인해줘” | 어떤 부분이 테스트 안 되는지 파악 |

“엣지 케이스(edge case)“란 극단적인 상황을 말합니다. 빈 값, 아주 큰 숫자, 특수문자 등 보통은 생각하지 못하는 경우입니다. 여기서 버그가 가장 많이 발생합니다.

---

## 정리

오늘 배운 것을 정리합니다:

- 테스트는 코드가 제대로 동작하는지 **자동으로 확인**하는 안전장치입니다

- Claude에게 “테스트 만들어줘”라고 하면 자동으로 생성해줍니다

- TDD 스타일로 테스트를 먼저 만들고 코드를 나중에 작성할 수 있습니다

- Hooks와 조합하면 커밋할 때 자동으로 테스트가 실행됩니다

- 테스트가 실패하면 Claude에게 고쳐달라고 하면 됩니다
